## 踩坑与解法
### networkidle 超时 refs: 2026-03-28-grubhub_menu_crawler
- 现象: Timeout 30000ms，SPA 持续后台请求
- 解法: 改用目标元素出现作为就绪信号

### Virtual Scroll 数据丢失 refs: 2026-03-28-grubhub_menu_crawler
- 现象: 滚动后 section 从 4 变 1
- 解法: 放弃 DOM，改为 API 拦截

### feed API 返回空结果 refs: 2026-03-28-grubhub_menu_crawler
- 现象: enhanced_feed 只是骨架
- 解法: 遍历点击分类 tab，每次 0.6s 间隔

### nodriver handler 注入失效 refs: 2026-03-28-grubhub_menu_crawler
- 现象: functools.partial 覆盖自动注入
- 解法: `async def handler(event, tab=None)`

### ResponseReceived body 未就绪 refs: 2026-03-28-grubhub_menu_crawler
- 现象: get_response_body 抛异常
- 解法: 两阶段拦截，LoadingFinished 才读 body

## 可复用模式
- 懒加载触发：点击 tab，0.6s 间隔
- 页面就绪：等待导航元素，不用 networkidle
- 两阶段拦截：ResponseReceived→ID，LoadingFinished→body
- 反爬：headless=False 效果最佳

Episodes 的双重作用： 一方面作为 daily-logs 的压缩摘要（减少 agent 扫描量）， 另一方面提取"知识信号"推动 ④ Knowledge 更新。 agent 历史查询时先搜 episodes（O(days) 复杂度），而非直接扫 daily-logs（O(events) 复杂度）。
Knowledge 层的两级结构： Memory.md 是跨领域的平面索引（每条一行摘要，<200行），供 agent 快速定位。 domain/*.md 是按领域划分的详细知识文件，包含类型层级 （project / feedback / reference / user）。 Memory.md 始终加载进上下文；domain 文件按需加载。

Agent vs LLM Call — 实现方式选择

T1/T2 选择 Agent 而非单次 LLM Call，有四个相互独立的真实理由： 命中缓存、按需加载、增量写入、行业验证。 T4 仅写快照无需读取，保持单次 LLM Call 即可。
触发点实现方式原因
T1 Independent Agent Independent Agent ① 命中缓存 — 加装主 agent session 后，消息前缀与主 agent 完全一致，Claude 服务端直接命中缓存，大幅降低 input token 成本。
② 按需加载 — 通过 MEMORY.md 索引定向读取相关 ④ knowledge 文件，无需全量预加载；知识库增长后仍保持高效。
③ 增量写入 — 使用 edit_file 只输出变更部分，无需输出整个文件覆盖；单次 LLM Call 全量覆盖写的 output token 成本随文件增长线性上升。
④ 执行一致性 — 复用主 agent 的 llmProvider、systemPrompt、toolCalls，提取行为与主 agent 完全一致，不存在"提取逻辑与主流程理解偏差"的问题。
⑤ 行业验证 — Claude Code、Hermes Agent、Open Claw 均采用 Agent 模式执行记忆提取，工程实践已验证。
T2 Main Agent Main Agent 执行路径与 T1 相同，同样具备以上四点优势；sync 执行，用户可见反馈
T3 Dream Task Agent (Cron) 跨文件聚合：读取全部 ②③④ 历史；Preference Promotion 需判断稳定性后动态路由至三个目标文件；文件数量不确定，必须动态检索
T4 Session Close 单次 LLM Call 只写 ② 快照，不读任何现有文件；单向转换（session 文本 → markdown 摘要）；速度优先，允许冗余，下次启动后由 T1 融合

T1 Independent Agent — Fork 实现方式

复用主 agent 的 llmProvider 和 systemPrompt，加装主 agent 的 session 消息作为上下文， 追加一条 user message 触发提取任务。主 agent 的 session 不受任何影响。

为什么复用 system prompt 且无需新建

主 agent 的 system prompt 已包含 workspace、instructions、memory 等完整上下文， memory agent 继承后直接具备提取所需的背景知识。无需另写专用 system prompt，减少维护成本。

为什么 Cache 真正命中

主 agent 持续向 Claude 发送 session 消息序列，Claude 服务端已按内容哈希缓存该前缀。 memory agent 加装相同 session 消息后，发送到 Claude 的内容前缀完全一致，直接命中缓存：

主 agent 已缓存: [system] [user₁] [asst₁] [user₂] [asst₂] ... memory agent 发送: [system] [user₁] [asst₁] [user₂] [asst₂] ... [user: 请提取记忆] ↑————————— 命中缓存 ——————————↑

执行流程

  • 1第 10 轮触发：快照 mainAgent.getMessages()（保留 SYSTEM，保留完整对话）
  • 2新建 Agent 实例，复用 mainAgent.llmProvider 和 mainAgent.systemPrompt，不设 persistenceProvider
  • 3memoryAgent.restoreHistory(snapshot) — 加装主 agent 全部 session 消息
  • 4memoryAgent.injectUserMessage("提取本次对话的记忆...") — 追加提取指令
  • 5独立线程异步执行 continueWithInjectedMessage()，主 agent 不感知、不阻塞
  • 6memory agent 执行提炼链，写入 ②③④；执行结果不回写主 agent messages
复用 system prompt 复用 session cache async 独立线程 不写回主 session

soul.md 与 CLAUDE.md 的关键区别：
soul.md 是 agent 的能力积累——"我学会了如何更好地解决这类问题"，偏向引导判断，语气是建议性的；
CLAUDE.md 是外部约束——"在这个项目里必须这样做"，偏向强制执行，语气是规定性的。
两者都常驻注入，但前者塑造思维方式，后者约束行为边界。

5实现要点 Implementation Notes

T1 触发条件检查

每 10 轮 main agent 在 PostToolUse hook 中检查以下条件（满足任一即触发）：

  • session 中 hot events 数量 ≥ 5
  • 存在 importance=high 的 hot event
  • 累计 issue 事件 ≥ 3 条
  • 距上次 T1 执行超过 30 分钟
注意：T1 使用独立 agent，不共享主 agent 的上下文窗口。通过文件传递 session 摘要。

MEMORY.md 维护规则

每次写入知识文件后需同步更新索引：

  • 每条索引 ≤ 150 字符，一行摘要
  • 总行数 < 200 行（超出则整理合并）
  • 格式：- [Title](file.md) — 简述
  • 删除指向不存在文件的索引行
MEMORY.md 在每次对话启动时自动加载到上下文，是知识系统的入口文件，保持精简至关重要。

6参考架构 Reference: AFU Memory System

来源：AFU Memory System — 三层架构，核心思想是将捕获（capture）与检索（retrieval）分离。 原始日志永不直接被查询，所有检索都走经过整理的语义知识层。这是本系统设计的重要参考依据。
图 5 — AFU Memory System 参考架构：三层分离（Real-Time Capture / Episodic / Semantic Knowledge）

核心设计原则（来自参考架构）

1. 捕获与检索分离

原始日志（current.jsonl）只负责写入，从不被直接查询。 检索入口永远是经过整理的 knowledge/ 语义层。 这避免了在噪声数据中搜索，也保证了检索精度。

2. JSONL 原子写入

实时捕获使用 .jsonl 格式（每行一个 JSON 对象），append-only。 Crash-safe：即使进程中途崩溃，已写入的行不会损坏， 下次恢复后仍可读取。比 Markdown 文件的实时写入更可靠。

3. 渐进式提炼管道

原始 → 叙事摘要 → 语义知识，三步渐进提炼。 每一步都有独立的存储位置和定时任务，互不阻塞。 这与本设计的 T3 Dream Task 思路完全一致。

4. 偏好提升是独立步骤

行为纠正（behavioral corrections）在当天发生，但不立即写入偏好文件。 每晚 23:50 单独评估：这是稳定模式还是一次性偏差？ 只有确认为稳定模式才提升到 preferences.md。

与本设计的映射关系

AFU 参考架构本设计对应差异与改进点
current.jsonl — 实时捕获 Session Hot Events（实时追加） 改进：采用 JSONL 格式替代 Markdown，提升 crash-safe 能力
daily-logs/ + episodes/ — 叙事摘要 ② .learning（T4 快速写入）→ ③ episodes（T1 整理后生成） 参考架构一步归档+合成；本设计拆分为 T4 快照写 ② / T1 独立提炼 ② → ③，层次更清晰
knowledge/ — 语义知识（检索入口） Domain Knowledge Layer（domain files） 结构一致；参考架构按 people/projects/decisions 分类，本设计按 project/user/feedback/reference 分类
preferences.md — 偏好提升 User Preference Layer（~/.claude/） 参考架构强调需经过稳定性验证才提升，本设计应加入此验证步骤
Nightly pipeline (23:30/23:45/23:50) T3 Dream Task (Cron 24h) 参考架构将三步拆为不同时间点，降低单次任务复杂度；本设计可拆分
watches/ — 监控目标 （本设计暂无对应） 可扩展：为特定项目/话题添加 watch 机制，追踪时序事件
关键设计决策（来自参考架构验证）： Session Hot Events 应使用 JSONL 格式（而非 Markdown 文件）进行实时捕获， 以保证 crash-safe 和 append-only 语义。Markdown 格式的整理版本（domain files / episodes） 作为人类可读的检索目标，在事后由 agent 生成。

查询时永远不直接搜索原始 JSONL 日志，只搜索经过整理的 knowledge 文件。
