2.3 Agent 执行流程

1
启动 forked agent

通过 runForkedAgent() 启动独立 agent，最多 maxTurns=5 轮

2
Turn 1: 并行读取

Agent 并行发起所有 FileRead 调用，读取需要更新的 memory 文件

3
Turn 2: 并行写入

Agent 并行发起所有 FileWrite/FileEdit 调用，写入/更新记忆

4
Turn 3-5: 可能的补充

如需额外操作（验证、更新索引等），最多再执行 3 轮

📐 Agent 执行流程

forked agent 通过 maxTurns=5 限制轮数，典型执行模式：

Turn 1
→

↓

↓
Turn 3-5

maxTurns = 5
  • 代码硬编码限制
  • 不可配置
  • 到达后强制结束
并行策略
  • Turn 1: 所有读取并行
  • Turn 2: 所有写入并行
  • 最大化效率

2.4 Agent 最大轮数限制

是代码逻辑强制限制，不是提示词。

提示词中只说"有限轮次预算"，没有具体数字。实际由代码层面的 maxTurns 参数控制：

2.5 用户通知机制

📐 通知策略

extractMemories 对不同情况的通知策略：

  • 告知用户记忆已保存
✗ 无记忆写入

3. autoDream (做梦) 机制

3.1 与 extractMemories 的核心区别

extractMemories

  • 触发: 每轮 post-sampling
  • 数据: 当前 session
  • 目的: 提取新记忆
  • 操作: 新增文件

autoDream

  • 触发: 时间 + 会话门控
  • 数据: 跨多个 session
  • 目的: 整合/清理已有记忆
  • 操作: 修改/合并/删除

3.2 触发门控（Gate 顺序）

📐 门控检查流程

门控按"最便宜优先"顺序检查，任何一步失败都会 return：

▶ 开始检查
↓
1 时间门控 hoursSince >= minHours (默认 24h)
❌ return
↓
2 扫描节流 距离上次扫描 >= 10 分钟
❌ return
↓
3 会话门控 sessionIds >= minSessions (默认 5)
❌ return
↓
✓ 执行 Consolidation
时间门控
  • 配置: minHours
  • 默认值: 24 小时
  • GrowthBook 可配置
扫描节流
  • 距离上次扫描 >= 10 分钟
  • 防止频繁扫描
会话门控
  • 配置: minSessions
  • 默认值: 5 个会话
  • GrowthBook 可配置

3.3 Dream 的三个阶段

Phase 1 — Orient (定位)
  • 列出 memory 目录
  • 读取 MEMORY.md 索引
  • 浏览现有 topic 文件
Phase 2 — Gather Signal (收集信号)
  • 读取 daily logs (每日日志)
  • 查找与现有记忆矛盾的信息
  • grep transcript 获取特定上下文
Phase 3 — Consolidate (整合)
  • 合并新信号到现有 topic
  • 将相对日期转为绝对日期
  • 删除/修正过时的记忆

4. 会话关闭处理

4.1 extractMemories: 有 drain 机制 ✓

📐 关闭流程（有 drain）

会话关闭时，extractMemories 会等待提取完成：

会话关闭
→
drainPendingExtraction
→
gracefulShutdown
✓ drainPendingExtraction
  • 等待最多 60 秒
  • 提取完成则继续
  • 超时后强制继续
gracefulShutdown
  • 5s failsafe
  • 进程退出
// extractMemories.ts:579-586
drainer = async (timeoutMs = 60_000) => {
  if (inFlightExtractions.size === 0) return
  await Promise.race([
    Promise.all(inFlightExtractions),           // 等待完成
    new Promise(r => setTimeout(r, timeoutMs)) // 或超时
  ])
}

4.2 autoDream: 无 drain 机制 ⚠️

📐 关闭流程（无 drain）

会话关闭时，Dream 直接被终止，不等待完成：

会话关闭
→
gracefulShutdown
→
Dream 被 kill
✓ 已写入的文件
  • 会保留
  • 不会回滚
↩️ Lock 文件
  • 会回滚 mtime
  • 下次可重新执行
⚠️ Consolidation 完整性
  • 可能不完整
  • 下次满足条件时重跑
情况 结果
已写入的文件 保留 不会回滚
Lock 文件 回滚 下次可重新执行

6. 总结

📐 双系统对比总结

extractMemories

  • 目的: 提取当前会话的新记忆
  • 触发: 每轮 post-sampling
  • 数据: 当前 session (完整历史)
  • Agent: forked, maxTurns=5
  • 节流: tengu_bramble_lintel
  • 通知: 有写入时显示
  • 关闭: drain 等待 60s ✓
  • 失败: 光标不更新，下次重试
VS

autoDream

  • 目的: 整合/清理/更新已有记忆
  • 触发: 时间(24h) + 会话(5) 门控
  • 数据: 多 session transcript
  • Agent: forked, 无 maxTurns 限制
  • 阶段: Orient → Gather → Consolidate
  • 通知: 有写入时显示
  • 关闭: 无 drain，立即终止 ⚠️
  • 失败: Lock 回滚，下次可重跑
